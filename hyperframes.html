01 · The premise

What HyperFrames actually is

One sentence: open-source HTML→MP4 renderer with built-in browser NLE, agent skills, and deterministic seek across animation runtimes. Tagline: "Write HTML. Render video. Built for agents."

Built by HeyGen. Inspired by Remotion (they credit it in source) — same headless-Chrome + FFmpeg approach. The split: Remotion = React TSX. HF = HTML + data-* attributes.

Why HeyGen open-sourced it

Render engine isn't the moat. Avatars + lipsync + voice cloning are. HF is bait — community ships blocks and skills; HeyGen sells avatars on top.

License is Apache 2.0 (OSI-approved). Use commercially at any scale. No per-render fees. No seat caps. Compare Remotion: source-available, paid above small-team thresholds.

03 · Surprise finding

Studio has a real NLE

First — what is "Studio"?

Studio = the browser-based editor that ships with HyperFrames. React app served at localhost:3002 when you run npx hyperframes preview.

Live preview iframe + multi-track NLE timeline + asset sidebar + code editor + render queue, all in one window. Tech: React 18/19, Zustand state, CodeMirror 6 editor, Tailwind, Vite. Source lives in packages/studio/ · README: packages/studio/README.md. Think CapCut Web meets a code editor. When this doc says "Studio", it means that app.

First read of the README suggested Studio was a preview-only tool. Source disagrees. packages/studio/src/components/nle/NLELayout.tsx is 458 lines. packages/studio/src/player/components/Timeline.tsx is 1,693 lines. This is closer to CapCut Web than to Remotion's preview pane.

05 · Schema

Core timeline types

From packages/core/src/core.types.ts. Read this once and the whole system clicks.

type TimelineElementType = "video" | "image" | "text" | "audio" | "composition";
type Orientation = "16:9" | "9:16";

CANVAS_DIMENSIONS = {
  landscape: { width: 1920, height: 1080 },
  portrait:  { width: 1080, height: 1920 },
};

interface TimelineElementBase {
  id, type, name; startTime, duration, zIndex; x?, y?, scale?, opacity?;
}

interface TimelineMediaElement extends TimelineElementBase {
  src, mediaStartTime?, sourceDuration?, isAroll?, volume?, hasAudio?
}

interface TimelineTextElement extends TimelineElementBase {
  content, color?, fontFamily?, fontSize?, fontWeight?,
  textShadow?, textOutline?, textHighlight?, textHighlightColor?, ...
}

interface TimelineCompositionElement extends TimelineElementBase {
  src, compositionId, variableValues?: Record<string, string|number|boolean>
}

type CompositionVariableType = "string" | "number" | "color" | "boolean" | "enum";
Strategic insight

Composition variables = template slots. We can build "Reel templates" with named slots like {title}, {ctaColor}, {logoSrc}, then stamp out hundreds of variants without LLM-generating HTML. This is how Videfy keeps token cost near zero on the free tier.

08 · Architectural insight

Determinism = byte-identical renders

HF's superpower. Same input → same output, every machine, every time. The mechanism is worth copying wholesale.

1
Paused timelines
GSAP / Anime / etc registered on window.__timelines with paused: true. Animations never play forward on their own.
2
Per-frame seek hooks
Engine calls timeline.seek(t) for each frame, then captures. The runtime is driven, not free-running.
3
Wall-clock banned
No Date.now, no requestAnimationFrame for animation logic. Lint enforces this.
4
Font data baked in
fontData.generated.ts ensures cross-machine glyph consistency.
5
Frame Adapter pattern
Every animation runtime exposes a seek(t) interface. New runtimes plug in without changes to the engine.
Why this matters for Videfy

Pipelines need determinism. Re-runs become idempotent. Caching works. A/B variants are cheap. Distributed render becomes possible (chunk frames, recombine).

12 · Snapshot pattern

Live-data workaround — fetch + freeze

"No fetch() during render" looks like a hard limit, but there's a clean escape. Snapshot data at render submit time, write to project, render reads it synchronously.

1
Pre-render hook
Videfy fetches API → writes data.json to HF project
2
HTML reads sync
Composition reads data.json via injected <script> at compile
3
Render is deterministic
Given the snapshot, the output is byte-identical
4
Re-render = re-snapshot
Idempotent — newer data, fresh render, no fragility

Better still: use HF composition variableValues. Already supported. Videfy fills variables, HF stamps the variant.

14 · The competition

How Videfy + Claude beat HeyGen

HeyGen — strengths + weaknesses

Strengths. 100+ stock avatars, custom avatar option, voice cloning, lip-sync, translation, dubbing, $89/mo Creator.

Weaknesses.

  • Avatars look uncanny corporate — nobody scrolls past one on TikTok
  • UI clicker, not agent — no "make 50 reel variants" workflow
  • Locked to their avatars, no NB2 / MJ / Veo / Sora
  • No real-footage + AI mix
  • B2B explainer focus, not reels / TikTok / IG
  • Per-seat pricing kills agencies + multi-account creators
  • API exists but locked-down + expensive
Videfy — what makes it win
  • Agent-native: script → assets → render → post in one prompt
  • Multi-engine gen: NB2, MJ, ChatGPT, Veo, Seedance, Kling, Sora
  • Real footage + AI mix — Pexels B-roll + AI gen + user assets
  • Editorial look — film stock, grain, no corporate avatar tells
  • Reels / TikTok native — 9:16, hook-first, scroll-stoppers
  • Pipeline composability — chain steps, schedule, batch
  • Open render layer (HF Apache 2.0), no per-render fees
  • BYOK pricing — bring your own keys, pay actual cost
  • Voice clone + lip-sync via OSS (ElevenLabs + Wav2Lip / LatentSync)
  • Skills marketplace: /videfy ecosystem
  • Multi-account creator + agency support — one workspace, N brands
  • Determinism enables overnight 100-variant batches
Positioning

"HeyGen for creators, not corporates."

Tagline: Type your reel. Get a reel. No avatars. ICP: creators 5K-100K followers, agencies running 5+ brands, content shops. Pricing: $29/mo Creator (BYOK), $99/mo Studio (managed gen + cloud render), $399/mo Agency (10 brands).

What HeyGen can't easily copy

Open-source positioning — their model = closed gen + locked avatars. Multi-engine gen — economics depend on selling their avatars. Editorial aesthetic — brand identity is corporate-explainer. Agent-native workflows — UI is template-clicker; rebuild = a new product.

# /videfy-gen

You generate AI imagery and short video clips for Videfy compositions.
Outputs MUST land in the active HF project's `assets/` directory.

## Resolution rules

1. Resolve the active HF project path from the conversation context.
   - User explicit path? Use it.
   - User in a `npx hyperframes preview` session? Read its project from `localhost:3002/projects`.
   - Otherwise ASK: "Which HF project should I write into?"

2. Aspect ratio defaults:
   - 9:16 portrait for reels / TikTok / Stories
   - 16:9 landscape for YouTube / explainers / web heroes
   - 1:1 for IG feed posts

3. Engine selection (default order):
   - Photorealistic editorial → ChatGPT (`gpt-image-2`) or NB2
   - Surreal / cinematic → Midjourney
   - Motion clip → Veo 3.1 (start frame from NB2 image)

## Tool usage

Always call `videfy-daemon` first via HTTP:

```
POST http://localhost:7777/generate/image
  { "prompt": "...", "engine": "nb2", "aspect": "9:16",
    "out": "<hf-project>/assets/<slug>-<n>.png", "n": 4 }
```

The daemon downloads the image bytes and writes to `out`. HF Studio
auto-refreshes the Assets tab via file watcher.

## Banned engines

NEVER use Flux. Stick to NB2, NB Pro, Midjourney, ChatGPT, Veo, Kling.

## Realism rules

- Film stock, grain, imperfection language
- No "beautiful" / "stunning" / "perfect" / "8K" / "hyperrealistic"
- No forced logos
- Always humans in frame (no empty still lifes)

## After generation

- Verify file exists on disk
- Report path back to user
- Suggest the next step: "Drag this from Assets tab to track 1 at t=0:00"
- Do NOT auto-edit the composition unless user requests

## When to escalate

If user asks for a full reel: hand off to `/videfy` orchestrator.
If user wants to wire into composition HTML: hand off to `/hyperframes`.

Mirror this structure for /videfy-script, /videfy-post, /videfy-stock, /videfy-clip, /videfy-shotlist. Each ~50-100 lines, frontmatter + body + tool usage + banned-paths + escalation.

B3 · Daemon code

Videfy daemon — Hono server

Local-first orchestrator on localhost:7777. Receives gen requests from skills, writes outputs to the active HF project, calls third-party gen APIs (Fal, Kie, OpenAI), returns absolute paths back. Stateless, deterministic, BYOK.

videfy-daemon/src/server.ts 
import { Hono } from "hono";
import { cors } from "hono/cors";
import { z } from "zod";
import { writeFile, mkdir } from "node:fs/promises";
import { dirname } from "node:path";
import { generateImage } from "./engines/index.ts";
import { generateVideo } from "./engines/video.ts";
import { synthVoice } from "./tts/kokoro.ts";
import { transcribe } from "./asr/whisper.ts";
import { searchBroll } from "./broll/pexels.ts";
import { postBlotato } from "./post/blotato.ts";

const app = new Hono();
app.use("*", cors({ origin: ["http://localhost:3002"] }));

const ImageReq = z.object({
  prompt: z.string().min(3),
  engine: z.enum(["nb2", "nb-pro", "mj", "chatgpt"]),
  aspect: z.enum(["9:16", "16:9", "1:1"]).default("9:16"),
  out: z.string(),       // absolute path inside HF project /assets
  n: z.number().int().min(1).max(8).default(1),
  refImages: z.array(z.string()).optional(),
});

app.post("/generate/image", async (c) => {
  const body = ImageReq.parse(await c.req.json());
  await mkdir(dirname(body.out), { recursive: true });
  const results = await generateImage(body);    // returns Buffer[]
  const paths: string[] = [];
  for (const [i, buf] of results.entries()) {
    const p = body.out.replace(/(\.\w+)$/, `_${i + 1}$1`);
    await writeFile(p, buf);
    paths.push(p);
  }
  return c.json({ ok: true, paths });
});

const VideoReq = z.object({
  prompt: z.string(),
  engine: z.enum(["veo-3.1", "kling-2.1", "seedance"]),
  startFrame: z.string().optional(),  // path to NB2 image
  duration: z.number().min(1).max(10).default(8),
  out: z.string(),
});
app.post("/generate/video", async (c) => {
  const body = VideoReq.parse(await c.req.json());
  await mkdir(dirname(body.out), { recursive: true });
  const buf = await generateVideo(body);
  await writeFile(body.out, buf);
  return c.json({ ok: true, path: body.out });
});

const TtsReq = z.object({
  text: z.string(),
  voice: z.string().default("af_heart"),
  out: z.string(),
});
app.post("/tts", async (c) => {
  const body = TtsReq.parse(await c.req.json());
  await synthVoice(body);
  return c.json({ ok: true, path: body.out });
});

const TranscribeReq = z.object({
  audioPath: z.string(),
  out: z.string(),  // writes word-level JSON
});
app.post("/transcribe", async (c) => {
  const body = TranscribeReq.parse(await c.req.json());
  await transcribe(body);
  return c.json({ ok: true, path: body.out });
});

app.post("/broll/search", async (c) => {
  const { query, source = "pexels", count = 10 } = await c.req.json();
  const results = await searchBroll({ query, source, count });
  return c.json({ ok: true, results });
});

const PostReq = z.object({
  videoPath: z.string(),
  caption: z.string(),
  accountIds: z.array(z.string()),
  scheduledTime: z.string(),  // ISO 8601
});
app.post("/post/blotato", async (c) => {
  const body = PostReq.parse(await c.req.json());
  const submissionId = await postBlotato(body);
  return c.json({ ok: true, submissionId });
});

// Health check
app.get("/healthz", (c) => c.json({ ok: true, version: "0.1.0" }));

export default { port: 7777, fetch: app.fetch };
videfy-daemon/src/engines/index.ts — engine router 
import { generateNB2 } from "./nb2.ts";
import { generateMJ } from "./mj.ts";
import { generateChatGPT } from "./chatgpt.ts";

export async function generateImage(req: {
  prompt: string;
  engine: "nb2" | "nb-pro" | "mj" | "chatgpt";
  aspect: "9:16" | "16:9" | "1:1";
  n: number;
  refImages?: string[];
}): Promise<Buffer[]> {
  switch (req.engine) {
    case "nb2":
    case "nb-pro":  return generateNB2(req);
    case "mj":      return generateMJ(req);
    case "chatgpt": return generateChatGPT(req);
  }
}
Why a daemon vs library

Daemon = language-agnostic. Skills call HTTP, not Node imports. Future Videfy clients (Python ML pipelines, n8n nodes, hosted web app) hit the same routes. Plus skills run inside Claude Code's process — calling external tools needs a network boundary anyway.

B9 · Token economics

Prompt cache strategy

Anthropic's prompt cache = 90% discount on repeated prefix tokens, 5-minute TTL on standard cache, 1-hour TTL on extended cache. Skills are big system prompts — caching them is the single biggest cost lever.

Rules

  1. Stable prefix. Skill bodies don't change between calls. Make them the first system block, mark cache_control: { type: "ephemeral" } on the last item of the cacheable section.
  2. Variables go after the cache cut. User prompt + composition state = post-cache. The cache key is the prefix hash.
  3. Keep skills under 2K tokens each. Bigger = pricier cold start. Move long examples into references/ and load on demand via tool call, not prefix.
  4. Use extended cache for the orchestrator. /videfy runs across long sessions — 1-hour TTL is worth the slight surcharge.
  5. Watch the metric. Token usage event includes cache_creation_input_tokens and cache_read_input_tokens. Read = 90% off. Creation = full price (one-time). Aim for cache_read > 95% of repeat calls within TTL.
